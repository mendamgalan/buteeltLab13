## I. Архитектур ба төлөвлөлт (Part A)

1) Project brief боловсруулах
- Copilot-оос хэд хэдэн төсөл саналыг авч, Personal Task Tracker-ийг сонгосон.
- Төслийн цар хүрээ, шаардлага, гол функцууд болон хугацааг тодорхойлсон.

2) Архитектурын шийдэл (ARCHITECTURE.md)
- Express дээр суурилсан layered архитектур (middleware → routes → controllers → data layer).
- SQL.js болон better-sqlite3-ийн давуу/сул талыг харьцуулсан.
- OpenAPI/Swagger-д зориулсан документацийн бүтэц төлөвлөсөн.

3) Stack шийдвэр (ADR-001)
- Node.js 20+, Express 4.x, SQLite (sql.js) хувилбаруудыг ашигласан.
- Zod валидаци, Jest + Supertest тестүүдэд ашигласан.

AI-гийн үүрэг: Stack сонголт, альтернатив саналууд, аюулгүй байдлын анхааруулга өгсөн.

## II. Хэрэгжүүлэлт (Part B)

### Code generation (гол хэсгүүд)

- Database: `db/database.js` болон `db/schema.sql` файлуудыг AI тусламжтай үүсгэж, persistence логик бий болсон.
- Route handlers: `routes/tasks.js`, `routes/labels.js` — CRUD, шүүлт/хайлтын боломжуудтай.
- Controllers: Бизнес логикыг тусад нь салгасан, алдааг барих try/catch ашигласан.
- Validation: `schemas/taskSchema.js` — Zod схемүүд (create/update/label).

### Тест ба дебаг

- Jest + Supertest ашиглан ~30+ тест бичигдсэн (happy path + алдааны тохиолдлууд), хамрах хүрээ ~75%.
- Гол засварууд: sql.js persistence (Buffer export), parameterized queries, төвлөрсөн error middleware, Zod validation, Swagger YAML засвар.

## VIII. What AI did vs What I did — Жишээ, нарийвчилсан тайлбар

Доорх хэсэгт төслийн тодорхой ажил дээр AI-гийн гүйцэтгэл ба миний (хүний) оролцоо хэрхэн ялгарсныг практик жишээгээр тайлбарлав. Энэ хэсэг нь AI-гийн өгсөн эх загварыг авч би юу, яагаад, хэрхэн өөрчилсөн талаар нарийвчлан харуулна.

1) Эндпоинтын скелет: `GET /api/tasks`
- AI хийсэн: Copilot тухайн маршрутын ерөнхий скелет, query параметрүүд (priority, labels, search) ба эхний SQL хэлбэр ("SELECT * FROM tasks WHERE 1=1")-г санал болгосон.
- Би хийсэн: AI-гийн саналыг аваад параметрийг `params` массивээр бинд хийх, SQL-инjection шалгалт нэмэх, `JOIN` ашиглан `task_labels`-оос шошго оруулах хэлбэрт өөрчилсөн. Мөн pagination (limit/offset) ба default сортлогдолтыг нэмсэн.
- Яагаад: Эхний код нь хурдан боловч N+1, paginationгүй, security-т сул байсан тул production-д ашиглахын өмнө сайжруулах шаардлагатай байв.

2) Database persistence (sql.js)
- AI хийсэн: AI `sql.js`-ийн buffer export-ыг жишээ кодоор харуулж, `fs.writeFileSync(dbBuffer)`-ыг санал болгосон.
- Би хийсэн: Би async вариант ашиглаж, process shutdown дээр buffer-г хугацаанд нь бичих логик нэмсэн, file lock болон write failure-кэйсийг харгалзан retry механизмыг нэмж, unit/integration тестээр проверк хийсэн.
- Яагаад: `writeFileSync` нь process-г саатуулж болох тул async/await болон try/catch-тай шийдэх нь илүү найдвартай.

3) Validation schemas (Zod)
- AI хийсэн: Zod схемийн ерөнхий бүтцийг генер хийсэн — `TaskCreateSchema` ба `TaskUpdateSchema`.
- Би хийсэн: Бизнес логикт таарах ямар constraint-ууд (жишээ: dueDate нь ирээдүйд байх, title-тай урт хязгаар) нэмсэн. Мөн labels массивын формат, уник байдал, баганын нэрүүдтэй нийцүүлэх шалгалтыг нэмж, мөн алдааны мессежийг хэрэглэгч-д ойлгомжтой болгосон.

4) Tests (Jest + Supertest)
- AI хийсэн: Happy-path тестүүдийг хурдан генерласан (create/read/update/delete).
- Би хийсэн: Edge-case тестүүд (validation fail, duplicate labels, concurrent updates), DB persistence test-үүд, мөн flaky тестээс зайлсхийсэн setup/teardown логикыг нэмж бичсэн.

5) OpenAPI/Swagger
- AI хийсэн: Анхны `openapi.yaml` саналыг өгсөн — маршрут, schema, status code-ууд.
- Би хийсэн: Response examples нэмсэн, 201 vs 200 ялгалтыг зөвлөн зассан, authentication схем (future) үр дүнгийн тэмдэглэгээг оруулсан.

Эдгээр жишээнүүд нь AI нь 'хүртээж авхаар' эхний загварыг маш хурдан гаргадаг, харин хүн нь контекст, аюулгүй байдал, performance, ба бизнес шаардлагыг харгалзан сайжруулдаг байдлыг тод харуулна.

## IX. Галлюцинаци (Hallucinations) — 2+ жишээ, нөхцөл байдал ба шийдэл

AI-тай ажиллахад хамгийн чухал эрсдлүүдийн нэг нь "галлюцинаци" — үүнээс үүдэн AI бодит бус, алдаатай эсвэл зохиомол мэдээлэл санал болгодог. Доорх 3 жишээг өгнө:

1) Жишээ 1 — Бүтэлгүй `sql.js` API нэр
- Нөхцөл: Би AI-д "export database buffer to file" гэж асууж, AI `db.exportToFile('tasks.db')` гэх мэт хялбар нэртэй API-г санал болгосон.
- Алдаа/галлюцинаци: `sql.js`-ын бодит API нь `db.export()` эсвэл `db.serialize()` маягийн функцуудтай бөгөөд `exportToFile` гэж байхгүй. Ийм кодыг шууд ажиллуулахад runtime алдаа үүссэн.
- Үр дүн: Туршилтаар алдаа гарч, CI build тасалдсан.
- Засарсан арга: Бид албан ёсны `sql.js` баримт бичгийг шалгаж, зөв API (`db.export()` → Buffer → `fs.writeFile`) ашигласан бөгөөд AI-гийн санал болгосон нэрийг зассан.

2) Жишээ 2 — Зөвшөөрөлгүй npm багц эсвэл буруу лицензийн мэдээлэл (галлюцинац)
- Нөхцөл: AI нэг удаа `fast-sqlite-helper` нэртэй хялбар npm багцыг санал болгосон ба энэ багц MIT лицензтэй гэж тайлбарласан.
- Алдаа/галлюцинаци: Энэ багц бодит ертөнцөд байхгүй байсан буюу лицензийн мэдээлэл буруу байсан (AI зохиомол нэр, эсвэл хуучин хувилбарын мэдээллийг будилуулсан). Түүнчлэн AI зарим кодын жишээг StackOverflow-ээс санаандгүй хуулбарласан байв.
- Үр дүн: Багаж ашиглах эсвэл кодыг шууд хуулж хариуцлагагүй лицензийн зөрчил үүсгэх эрсдэлтэй.
- Засарсан арга: Бид багцын бодит байдал, лицензийг `npm view <package> license` болон GitHub хаягаар шалгасан. Any third-party snippet-ыг ашиглахдаа эх сурвалж, лиценз бүхий зохих зааврыг мөрдсөн.

3) Жишээ 3 — OpenAPI төрөл (type) галлюцинаци
- Нөхцөл: AI `openapi.yaml`-д `dueDate`-ыг `integer` болгон үзүүлсэн (UNIX timestamp гэж тааварласан) гэвч бидний код ISO 8601 strings-ийг хүлээн авч байв.
- Алдаа: Энэ зөрүү тестэд алдаа үүсгэсэн — клиент төрлөө буруу илгээсэн тул API validation-г дамжуулсангүй.
- Засарсан арга: API схемийг бодит хэрэглээтэй нийцүүлж, docs-ыг кодтой синхрончилсан.

Эдгээр жишээнүүд нь AI-гийн санал болгосон бүх зүйлийг автомат итгэхгүй, эх сурвалж, баримт бичгийг шалгах хэрэгтэй гэдгийг харуулна.

## X. Security / License жишээ

Нэг тодорхой практик жишээ:

- Нөхцөл: AI нэгэн удаа `dbPath`-ыг environment variable-аас шууд авч, хэрэглэгчийн өгсөн `DB_PATH`-ийг код дотор `fs.readFileSync(process.env.DB_PATH)`-ээр уншиж хэрэглэх саналыг өгсөн.
- Аюул: Хэрэв `DB_PATH` нь хэрэглэгчийн хяналтан дор байна (жишээ нь файл upload эсвэл untrusted input), энэ нь path traversal, arbitrary file read/overwrite, эсвэл sensitive файлын илрүүлэлтэд ашиглагдаж болно.
- Би авч хэрэгжүүлсэн шийдэл: Би `DB_PATH`-ыг configuration file-д эсвэл `.env`-д зөвшөөрөгдсөн path-уудаар хязгаарлаж (whitelist), path canonicalization-ыг ашиглаж, runtime-д `fs.stat`-гаар зөвшөөрөгдсөн файлуудыг шалгасан. Мөн DB buffer-ийг бичихдээ temporary file -> atomic rename утгатай (`fs.writeFile` ба `fs.rename`) ашигласан.

Лицензийн талаарх сануулга:
- Нөхцөл: AI заримдаа кодын жишээг StackOverflow эсвэл бусад эх сурвалжаас хуулбарлаж, эх сурвалжийг дурдаагүй тохиолдол байсан.
- Аюул: Энэ нь лицензийн зөрчил үүсгэж, сурвалжийн нөхцлийг зөрчих эрсдэлтэй.
- Би хийсэн: Бүх AI-гаар үүсгэсэн кодын жишээнүүдийг manual review-д оруулж, шаардлагатай бол эх сурвалж зааж өгөх, мөн лицензийн багцыг шалгаж, шаардлагатай тохиолдолд өөрчлөлт хийсэн.

## XI. Ур чадвар алдалтын (skill atrophy) эсрэг стратеги — дэлгэрэнгүй арга хэмжээ

AI нь үр ашигтай боловч ур чадвар алдалтыг үүсгэх эрсдэлтэй. Бид дараах практик, хэмжигдэхүйц стратегийг хэрэглэсэн ба санал болгож байна:

1) Албан ёсны "No-AI" цагууд
- Долоо хоногт 2 цагийг AI ашиглахгүйгээр код бичихэд зарцуулна. Энэ хугацаанд жижиг функц, багц, эсвэл refactor-ыг гардан бичнэ.

2) Code katas ба алгоритмын дасгал
- Савласан (20-30 минут) алгоритмын дасгалуудыг регуляр хийж, бодол бодож шийдэх дадалтай байна. Жишээ: массив-сум, string манипуляци, SQL query оптимизаци гэх мэт.

3) Explain-to-peer (сурган тайлбарлах)
- Оюутан болон багийн гишүүд 1 удаа долоо хоногт 15–30 минутын турш өөрсдийн хийсэн кодыг тайлбарлана. Тайлбарлах үед AI-г ашиглахгүй.

4) TDD-first (tests-first) практик
- Жижиг feature-үүдийг эхлээд тест бичиж (микротест), дараа код бичнэ. Ингэснээр таниулах чадвар болон ойлголт бий болно.

5) Periodic "AI audit"
- Хүснэгт: 1 удаа долоо хоногт AI-гаар үүсгэсэн commit-үүдийг review хийж, аль хэсгийг AI-гээс авсан, аль хэсгийг хүн сайжруулсан гэдгийг тэмдэглэнэ. Энэ нь AI-ийн тусламжтайгаар хийсэн чөлөөтэй талуудыг тодорхойлодог.

6) Restricted sandbox challenges
- CI дээр "no-AI" branch-аар албан сорил өргөж, тухайн branch-д AI-гүйгээр feature хийхээр шалгана.

7) Мэргэжлийн хөгжлийг дэмжих
- Санал: Code review-д human mentor-ууд оруулах, алгоритм болон архитектур дээр семинар зохион байгуулж, AI-гаар хурдан хийх боломжтой даалгавруудад гараар илүү төвлөрөх.

Эдгээр арга хэмжээнүүдийг тогтмол мөрдөх нь AI-гаар хамааралтай байдал бууруулж, баазаар ур чадварыг хамгаалах тусгай практик юм.

## XII. Нэмэлт сургамж ба практик зөвлөмжүүд (Expanded Lessons)

- Prompt engineering бол ур чадвар: Тодорхой, контекст олгож, "constraints"-г зааж өгөх нь AI-гийн үр дүнг ихээхэн сайжруулна.
- Always treat AI output as draft — unit/integration tests-ээр шалгахгүйгээр хэзээ ч production руу шууд оруулахгүй.
- Create a short verification checklist: `security`, `performance`, `license`, `tests`, `docs` — AI-гийн санал бүрд энэ шалгалтыг хийх.
- Keep an approved package list and a denied package list (security/licensing reasons). AI санал болгосон third-party packages-г энэ жагсаалттай шууд харьцуул.
- Train team on spotting hallucinations: API method names that "feel" wrong, invented package names, implausible performance claims.
- Use AI to generate tests as well as code: asking AI to create failing tests first helps catch hallucinations early.

## XIII. Нийт дүгнэлт

AI-тай хамтарсан энэ төсөлдээ бид богино хугацаанд өндөр үр дүнд хүрч чадсан. AI нь boilerplate, документац, тестийн skeleton-ыг хурдан гаргаж өгсөн бол хүний оролцоо нь кодын найдвартай байдал, аюулгүй байдал, performance болон бизнес шаардлагыг хангах гол хүч болсон. Галлюцинацийг илрүүлэх, лиценз, security асуудлыг шалгах, ур чадвар алдалтыг урьдчилан сэргийлэхэд тодорхой процесс, сургамж шаардлагатай.

Хэрвээ та энэ өргөтгөсөн тайлбар дээр нэмэлт болгох, тодруулах зүйл байвал хэлнэ үү — би баримт бичгийг илүү тодорхой, эсвэл англи хэл дээр доплан болгож өгч болно.

***

- AI нь хурдан boilerplate, тест, документац үүсгэхэд маш үр дүнтэй.
- Гэхдээ production-д оруулахын өмнө хүн гардан шинжилгээ, security audit хийх шаардлагатай.
